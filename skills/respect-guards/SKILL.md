# Respect Guards

Manage guard scripts that prevent repeating past mistakes. Guards are generated by `/oops` and live in `~/.claude/respect/guards/`.

## Commands

### List guards

1. Scan `~/.claude/respect/guards/*.json` for trigger metadata files
2. For each `.json` file, extract: triggers, lesson, created date
3. Check if corresponding `.sh` file exists and is executable
4. Display in table format:

```
Guards
======
| Guard | Triggers | Lesson | Status |
|-------|----------|--------|--------|
| no-timeout | timeout | Use gtimeout on macOS | active |
| no-friday-deploy | git push, deploy | Don't deploy on Fridays | active |
| broken-guard | rm -rf | Be careful | missing .sh |
```

If no guards exist, display: "No guards installed. Guards are created automatically when you use /oops."

### Fix a guard

When the user says "fix the [name] guard" or reports a guard misfiring:

1. Read both `~/.claude/respect/guards/[name].sh` and `[name].json`
2. Understand what the guard is trying to prevent (from the lesson and trigger metadata)
3. Ask the user what went wrong (false positive? missed a case? wrong trigger?)
4. Rewrite the guard script and/or trigger file to fix the issue
5. Make the script executable: `chmod +x [name].sh`
6. Confirm: "Updated guard [name]. It will now [description of new behavior]."

### Delete a guard

When the user asks to remove/delete a guard:

1. Confirm: "Delete guard [name]? This will stop checking for [lesson]. (y/n)"
2. On confirmation, remove both `[name].sh` and `[name].json` from the guards directory
3. Confirm: "Guard [name] deleted."

### Test a guard

When the user asks to test a guard:

1. Read the guard script and trigger file
2. Run the guard with a sample tool input that should trigger it:
   `bash ~/.claude/respect/guards/[name].sh "Bash" "[sample input matching triggers]"`
3. Show the result: exit code and output
4. Run with a sample that should NOT trigger it
5. Report: "Guard [name] fires correctly on [trigger] and passes on [non-trigger]."

## Important

- Guard files are always in `~/.claude/respect/guards/`
- Each guard has a `.sh` (executable script) and `.json` (trigger metadata) file
- The `.sh` script receives `TOOL_NAME` as `$1` and `TOOL_INPUT` as `$2`
- Exit 1 with a message = warn the user. Exit 0 = allow the action.
- The `.json` `tools` field (optional) controls which tools the guard applies to:
  - Omitted = defaults to Bash/Write/Edit/NotebookEdit
  - Regex patterns like `mcp__.*slack.*` target specific MCP tools
  - Can mix: `["Bash", "mcp__.*datadog.*"]`
- Never modify guard scripts without reading them first
